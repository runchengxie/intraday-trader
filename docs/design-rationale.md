# 设计思路

本文记录 intraday-trader 的策略设计、风控模型、绩效评估框架和多券商选型背后的设计考量。这些内容最初来自课程研究背景，现已提炼为项目自身的设计说明。阅读本文有助于理解设计考量，而不是只看当前实现。

当前实现状态以 `docs/project-manual.md` 和代码为准。

## 策略选型

### 为什么是均值回归 + 趋势跟随

项目同时纳入均值回归和趋势跟随两类策略，初衷是在同一套数据和风控框架下对比两种截然不同的收益来源：

- 均值回归假设价格在短期偏离后会回归中枢，适合震荡市
- 趋势跟随假设价格会延续已有方向，适合趋势市

两类策略在不同市场状态下的表现天然互补。在同一回测框架中跑两类策略，可以观察它们在相同时间段内的收益相关性、回撤时机和换手特征，避免只盯着单一逻辑做过度优化。

### 具体策略选择

均值回归：Z-Score 滚动标准化

用滚动窗口的均值和标准差将当前价格映射为 Z-Score。Z-Score 超过阈值时触发反向信号。选择 Z-Score 的考虑：

- 计算简单，参数少（只有窗口长度和阈值），不容易过拟合
- 标准化后的信号在不同价格水平和波动率下可比
- 支持 `filtered_close` 输入，可以在信号生成前对价格序列做预处理

Z-Score 的局限也很明显：它假设价格分布是平稳的，在趋势市中会持续产生错误信号。这是均值回归策略的固有特征，不是 bug。

趋势跟随：EMA 交叉 + ADX 过滤

短周期 EMA 上穿长周期 EMA 做多，下穿做空。加入 ADX 过滤的考虑：

- 纯 EMA 交叉在震荡市中假信号太多
- ADX 衡量趋势强度，不判断方向，可以和 EMA 交叉的方向信号正交组合
- 移动止损用来在趋势反转时保护利润

趋势跟随补充：价格与长期均线比例

`CustomRatioStrategy` 用当前价格除以长期均线，比值大于 1 偏强，小于 1 偏弱。这个策略的定位是提供一个更简单的趋势基准。如果加了 ADX 和移动止损的 EMA 策略没有显著优于这个简单比例策略，那复杂度可能不值得。

### 买入持有基准

基准策略有两个角色：

1. 判断主动策略是否真的贡献了 alpha（而不是跟着市场涨）
2. 支持含股息总回报的计算（在 Alpaca 可返回股息数据时），避免低估基准收益

## 风控模型设计

### 分层检查，逐级拦截

风控分两层，各司其职：

1. 行情层：`RiskManager.update_market_data()` 在每次行情更新时检查价格跳变、成交量异常和流动性异常。这些检查频率高、计算轻，在事件循环内完成。

2. 下单层：`check_liquidity_and_impact()` 和 `check_leverage_and_exposure()` 在下单前执行。这些检查涉及当前持仓和账户状态，只在实际要下单时才需要。

分层设计的意图：行情层检查是常态防御，拦截数据异常和极端行情；下单层检查是执行关闸，阻止在不利状态下增加风险敞口。两层之间没有依赖，任一层拦截都能阻止订单。

### 参数选择思路

| 参数 | 设计意图 |
| --- | --- |
| `max_order_participation_ratio`（默认 0.02） | 防止单笔订单占比过高，在流动性不足的时段推高成交成本。0.02 是保守值，大市值标的可以适当放宽 |
| `max_bid_ask_spread_pct`（默认 0.005） | 0.5% 的价差阈值在美股中等流动性标的上偏宽松，目的是在正常时段不阻止交易，仅在价差异常扩大时拦截 |
| `market_impact_coefficient`（默认 0.5） | 线性冲击估算的系数。这是简化模型，真实市场冲击是非线性的。0.5 是偏保守的估计 |
| `max_gross_exposure`（默认 1.5） | 允许多空同时持仓但限制总敞口。1.5 意味着多空各 75% 仓位可以共存 |
| `max_leverage`（默认 2.0） | 允许适度杠杆，但在 Alpaca 纸上交易环境中实际杠杆通常受限于账户类型 |
| `max_concentration`（默认 0.3） | 单标的 30% 上限，在单标的策略下意义不大，为多标的扩展预留 |

这些默认值是为纸上交易联调设计的偏保守值。实盘前应根据标的流动性、账户规模和实际交易成本重新校准。

## 绩效评估框架

### 为什么不只看夏普

`PerformanceAnalyzer` 同时输出多个指标，因为单个指标各有盲区：

- 夏普比率假设收益正态分布，日内策略的收益分布往往有肥尾
- 最大回撤只看最差情况，不反映回撤频率和持续时间
- 胜率不反映盈亏金额，高胜率低盈亏比的策略可能不如低胜率高盈亏比

所以输出清单里包含了 VaR、CVaR、换手率和交易成本。这些指标从不同角度约束策略质量：

- VaR 和 CVaR 捕捉尾部风险
- 换手率衡量策略的实际交易强度（高换手可能意味着交易成本吃掉 alpha）
- 交易成本直接计入绩效，避免把 gross alpha 当 net alpha

### 因子归因的缺失（已知局限）

当前没有内置因子回归模块。这意味着回测收益中可能包含市场 beta、规模因子或动量因子的贡献，无法直接拆出纯 alpha。

如果后续要做策略归因，建议在 `PerformanceAnalyzer` 或新模块中加入：

- 滚动 beta（策略对市场指数的滚动暴露）
- 信息比率（相对基准的超额收益除以跟踪误差）
- 多因子回归（市场、规模、价值、动量等因子的收益分解）

## 数据质量设计

### 缓存层级

三级缓存的意图是分层解决不同问题：

| 层级 | 后端 | 用途 |
| --- | --- | --- |
| 内存 | dict | 同一进程内重复查询时的瞬时访问 |
| 本地 | SQLite / Parquet | 单机持久化，快速回测和开发 |
| 远程 | PostgreSQL / TimescaleDB | 多进程共享、长历史存储、仪表盘查询 |

### 字段降级策略

`trade_count` 和 `vwap` 不是所有数据源都默认提供。项目采用可降级、可要求的策略：

- 默认：缺字段时回测自动估算（用简单均价代替 VWAP，用默认成交量代替未知 trade_count）
- 严格模式：设置 `data.require_full_fields: true` 强制拒绝不完整数据

这种设计避免了两种极端：一有缺字段就崩溃（开发体验差），以及永远不检查字段完整性（数据质量不可控）。

## 多券商与多行情源设计

### 为什么引入统一协议层

最初项目只有 Alpaca 一个券商通道，券商调用和行情拉取直接使用 `alpaca-trade-api` 的原生对象。接入富途后，直接依赖原生对象会带来两个问题：

1. 策略层和风控层需要知道当前用的是哪个券商，根据对象类型走不同分支
2. 每加一个新券商，所有下游代码都要改

统一协议层的方案是：每个券商适配器对外只返回 `StandardAccount`、`StandardPosition`、`StandardOrder` 这些数据类，行情源只返回包含标准列的 DataFrame。策略和风控代码永远通过协议层工作，不感知底层是 Alpaca 还是富途。

### 工厂模式

券商和行情源都通过工厂函数创建：`create_broker(app_config)` 和 `create_data_provider(app_config)`。工厂读取 `config.yml` 中的 `name` 字段决定走哪个适配器，未配置时回到 Alpaca 以保持向后兼容。

### 执行管线的定位

执行管线（`execution/`）填补了策略信号到券商下单之间的空白。原来的代码直接在交易循环里构造下单参数然后调用 `BrokerAPIHandler.place_order()`，缺少中间层来标准化信号转换、数量取整和限价偏移处理。引入管线后：

- `signal_to_target()` 把策略信号转为标准化的持仓目标
- `build_order_plan()` 把目标转为包含具体数量、价格和订单类型的下单指令
- `execute_order_plan()` 通过券商适配器提交指令

管线本身不持有券商引用，由调用方传入。这样同一个管线可以对接 Alpaca、富途或任何满足 `BrokerAdapter` 协议的后端。

### QEE 路由的定位

QEE 路由（`qee_execution_backend.py`）是执行的另一种出口：信号不直连券商，而是导出为 `targets.json` 再由 `quant-execution-engine` 处理。适用于需要审计追踪、多步风控审批或统一执行网关的场景。当前阶段 QEE 后端支持 dry-run 回放和实盘提交两种模式。

### 富途适配器当前局限

- 实时行情目前走 REST 轮询，未接入 WebSocket 推送。Alpaca 和富途的推送模型差异较大（Alpaca 用 WebSocket JSON 事件流，富途用 TCP 二进制协议），统一流式协议放在后续阶段。
- 订单类型当前只支持市价单和限价单（富途侧映射为 `OrderType.MARKET` 和 `OrderType.NORMAL`），不支持止损单等复杂订单类型。
- 富途 SIMULATE 环境的成交模拟依赖 FutuOpenD 自身的行为，成交率和延迟不受本项目控制。

## 技术选型

| 选择 | 理由 |
| --- | --- |
| Backtrader 回测框架 | 事件驱动架构天然接近实盘执行逻辑，回测到实盘的策略迁移成本低 |
| Python 3.10 锁定 | 回测框架的依赖兼容性和行为可复现性优先于语言新特性 |
| SQLite 默认存储 | 零配置、单文件、可移植，适合本地开发和快速迭代 |
| TimescaleDB 可选 | 为长历史和仪表盘查询提供时间序列优化，但不强制 |
| Docker Compose profiles | 按需启动数据库或交易服务，避免只是为了 dry-run 就拉起来一整个小型文明社会 |
| asyncio 事件循环 | 实盘端需要同时处理 WebSocket 行情推送和 REST 订单操作，异步模型比多线程更适合 I/O 密集型任务 |
| 统一协议 + 工厂模式 | 券商和行情源可插拔切换，策略和风控代码不绑定具体通道 |
